Chekin Guest SDK · design prototype

A check-in widget that behaves like a guest
on someone else's website

The guestapp redesign (one folder up) is expressive — gradients, glass, a navy shell. The SDK is the same product embedded in a partner's page, so this prototype goes the other way: the neutral slate palette of the hosts-sdk, the host's system font, no photography, and depth from borders instead of shadows. Same flows, different manners.

PrinciplesEvery screen in this prototype follows six rules — they're what make it an SDK design rather than a shrunken app.

A guest of the host page

System font stack (no webfont download forced on the partner), one muted accent (#505077, from the hosts-sdk), no hero images. The partner's brand stays the loudest thing on the page.

Bounded, always

Modals, splashes, spinners and crashes all render inside the widget's box (the real SDK's iframe). Nothing ever paints over the host page — see the errors page for the proof.

Border-first depth

No gradients, no liquid glass, whisper-quiet shadows. Hierarchy comes from 1px strokes, spacing and type weight — the language of the dashboard kit, not the guestapp's.

Themeable by CSS variables

The mock partner page carries Nordica's green brand. Flip the Widget accent toggle on any flow page and the widget adopts it — one repointed variable, standing in for the SDK's styles / stylesLink injection. Theming is a re-point, never a fork.

Events out, config in

The dark console under each frame logs the ChekinPro callbacks (onGuestRegistered, onIVFinished, onError…) as you click through, and each page's footnote names the config options that shape it.

Shared bones with the guestapp

Same semantic components (checklist row, field, option card, capture frame, mini-steps) and same token names — only surface values differ. That's the dashboard's shared-package recipe applied here.

Where this points the codebase

Today guestapp and SDK are one build with a flag (VITE_APP_IS_SDK) — one design serves two very different contexts. The dashboard repo shows the alternative: apps stay thin, while @chekin/shared-style packages own components, hooks and providers, and @chekinapp/tokens ships base values plus one surface file per app. This prototype is drawn so that split works: every screen here uses the same semantic components as the guestapp redesign, restyled purely by surface tokens (guestapp.css vs a new guest-sdk.css).
